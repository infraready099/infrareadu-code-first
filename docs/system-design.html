Section 1

The Big Picture

InfraReady uses two separate AWS accounts. Understanding this split is the most important thing about our architecture. One is ours — we run the software. One is the customer's — their actual infrastructure lives there.

The core promise: When a customer uses InfraReady, their database, servers, and networking all live inside their own AWS account — not ours. We never touch their data. They can cancel InfraReady tomorrow and keep everything running. This is what "Bring Your Own Cloud" means.

Section 2

The 5 Systems We Built

InfraReady is made of five distinct systems. Each one has a specific job. Here's what each one is and why it exists.

System A
The Runner (Lambda + SQS)
This is the engine of InfraReady. When a customer clicks "Deploy", a job goes into our SQS FIFO queue. Lambda picks it up (only one at a time, to avoid race conditions), temporarily borrows the customer's AWS credentials via STS, then runs tofu init → plan → apply for all 11 infrastructure modules. The Lambda has a 15-minute timeout — enough for even complex deployments.
AWS Lambda SQS FIFO OpenTofu 1.9.0 Node.js 20 15 min timeout
System B
The Frontend (Next.js on Vercel)
The web app customers use. Built with Next.js 15, deployed on Vercel. It handles auth (GitHub OAuth via Supabase), the 3-step onboarding wizard, and real-time deployment logs. API routes on the same app handle all backend logic.
/ Redirects — to /login if signed out, /projects if signed in
/login GitHub OAuth sign-in via Supabase
/projects Customer's list of all their InfraReady projects
/projects/new 3-step wizard: connect GitHub repo → connect AWS → configure and deploy
/projects/[id] Real-time deployment logs streaming from Supabase
/api/aws/connect Verifies the customer's IAM role is set up correctly via STS
/api/deploy Sends the deploy job to SQS queue
/api/github/connect + callback + webhook GitHub App OAuth flow and incoming push event webhooks
Next.js 15 Vercel Supabase Auth TypeScript
System C
The Bootstrap Role (CloudFormation)
A customer runs exactly one CloudFormation stack in their AWS account — this is the only manual step they ever do. It creates an IAM role called InfraReadyRole. This role is locked down: only our AWS account (073941171096) can assume it, and only when they present a secret ExternalId that is unique to each customer. This prevents "confused deputy" attacks — even if someone knew our account ID, they couldn't access your infrastructure without your ExternalId.
Security note: The ExternalId is generated uniquely per customer when they sign up and stored in our Supabase database. Our Lambda sends it with every STS AssumeRole call. Without it, the role cannot be assumed — even by us.
CloudFormation IAM Role ExternalId Protection STS AssumeRole
System D
GitHub Integration
We run a GitHub App called InfraReady-app (ID: 3061707). Customers install it on their repository during the wizard. After installation, every push to their main branch sends a webhook to our /api/github/webhook endpoint. InfraReady automatically queues a new deployment — so their infrastructure stays in sync with their code.
GitHub App #3061707 Webhooks OAuth Flow Installation Tokens
System E
CI/CD Pipeline (GitHub Actions)
Every time we push code to packages/runner/**, GitHub Actions automatically builds a fresh Docker image containing Node.js 20 and the OpenTofu 1.9.0 binary. It authenticates to AWS using OIDC — no stored AWS keys anywhere in GitHub. The image gets pushed to our ECR registry and Lambda is updated automatically. Our compliance pipeline also runs on every push: it validates all 11 OpenTofu modules, runs tflint (linter), Checkov (security scanner), and our SOC2/HIPAA policy checks.
GitHub Actions OIDC (no stored keys) ECR Checkov (security) tflint Docker
Section 3

The Deployment Flow

From sign-up to running infrastructure — here is every step in order, and who is responsible for each one.

1
Customer signs up with GitHub OAuth
They visit infraready.io, click "Sign in with GitHub". Supabase handles the OAuth flow — no password, no friction. Their GitHub identity is stored in our Supabase users table.
GitHub OAuth → Supabase Auth → User record created
2
Customer runs the CloudFormation bootstrap in their AWS account
Step 2 of the wizard shows a link to a CloudFormation template hosted in our public S3 bucket. They click it, AWS opens the stack creation page, they click "Create Stack". This creates the InfraReadyRole IAM role. This is the only AWS action they do manually.
bootstrap-role.yaml → InfraReadyRole with ExternalId trust policy
3
Customer connects their GitHub repository
They install the InfraReady GitHub App on their chosen repository. Our app gets webhook access and can read repository contents. We store the installation ID so we can generate access tokens later.
GitHub App installation → webhook configured → installation token stored
4
Customer fills the wizard and clicks Deploy
The wizard collects: their AWS account ID, the role ARN, their AWS region, and which modules they want enabled. Before they click Deploy, our /api/aws/connect endpoint does a test STS AssumeRole to verify the IAM role exists and is correctly configured.
STS test call → role verified → project record created in Supabase
5
Frontend sends a deploy job to SQS
The /api/deploy API route packages up all the deployment configuration (account ID, role ARN, ExternalId, region, modules) into a message and sends it to our SQS FIFO queue. The deployment record is written to Supabase with status "pending".
SQS SendMessage → deployment row in Supabase (status: pending)
6
Lambda picks up the job from SQS
Our Lambda function is subscribed to the SQS queue with a batch size of 1 — it processes one deployment at a time. Lambda marks the deployment as "running" in Supabase. The 15-minute execution window begins.
SQS trigger → Lambda invoked → deployment status: running
7
Lambda calls STS AssumeRole to enter the customer's AWS account
Lambda calls AWS STS with the customer's role ARN and their unique ExternalId. STS returns temporary credentials (access key, secret key, session token) that are valid for the duration of the deployment. Lambda now has permission to create resources inside the customer's AWS account.
sts:AssumeRole → temp credentials (15 min) → scoped to customer account
8
Lambda runs OpenTofu across all 11 modules
Using the temporary credentials, Lambda executes tofu init, tofu plan, then tofu apply --auto-approve for each module in order: first VPC (networking must exist before everything else), then RDS, ECS, Storage, Security, WAF, KMS, Backup, Inspector+SSM, Macie, and finally VPC Endpoints.
tofu init → plan → apply × 11 modules (sequenced by dependency)
9
Logs stream to Supabase in real time
As each OpenTofu command runs, Lambda writes log lines into the deployment_logs table in Supabase. The frontend page at /projects/[id] subscribes to Supabase Realtime — the customer sees logs appearing live, line by line, like a terminal output.
Lambda → Supabase deployment_logs → Realtime subscription → UI
10
Done — customer's AWS account has full production infrastructure
Lambda marks the deployment as "success" in Supabase. The frontend shows a completion screen. The customer's AWS account now has: a private VPC, a managed PostgreSQL database, containerized app hosting with auto-scaling, a CDN, a WAF, encryption keys, audit logging, automated backups, and vulnerability scanning — all in about 20 minutes.
deployment status: success → 11 modules live → customer owns all of it

Section 4

The 11 OpenTofu Modules

Each module is a self-contained piece of infrastructure. Together they form a complete, SOC2-ready production environment. Every module passes our CI compliance checks on every push.

VPC CI Pass
Private cloud network. Subnets (public + private), NAT gateway, internet gateway, route tables. All other modules deploy inside this VPC.
RDS CI Pass
Managed PostgreSQL database. Multi-AZ, automated backups, encryption at rest, parameter groups, subnet groups, and security groups.
ECS CI Pass
Container hosting. ECS Fargate cluster, Application Load Balancer, auto-scaling policies, task definitions, ECR repository, and health checks.
Storage CI Pass
S3 bucket for static assets and uploads. CloudFront CDN distribution with custom domain support, Origin Access Control, and cache behaviors.
Security CI Pass
CloudTrail (audit logs), CloudWatch (metrics and alarms), AWS Config (resource compliance rules), GuardDuty (threat detection). SOC2 baseline.
WAF CI Pass
Web Application Firewall. AWS Managed Rule Groups: Core rules, Known Bad Inputs, IP reputation lists. Attached to the CloudFront distribution and ALB.
KMS CI Pass
AWS Key Management Service. Customer-managed encryption keys for RDS, S3, and CloudWatch Logs. Automatic key rotation enabled.
Backup CI Pass
AWS Backup. Automated backup plans for RDS and EFS. Configurable retention periods, cross-region copy rules, and vault lock policies.
Inspector + SSM CI Pass
Amazon Inspector v2 for vulnerability scanning of EC2 and container images. SSM Session Manager for secure shell access without opening SSH ports.
Macie CI Pass
Amazon Macie. Automatically discovers and protects sensitive data (PII, credentials, financial data) stored in S3. Required for HIPAA-adjacent compliance.
VPC Endpoints CI Pass
Private connectivity to AWS services (S3, DynamoDB, Secrets Manager, SSM, ECR, and more) without traffic leaving the VPC. Reduces data transfer costs and eliminates exposure to the public internet for internal AWS API calls. Gateway and interface endpoint types.
All 11 modules pass CI on every push. Our compliance pipeline runs OpenTofu validate, tflint (code quality linter), Checkov (security scanner against 500+ rules), and custom SOC2 and HIPAA policy checks. Results appear in the GitHub Security tab as SARIF reports.

Section 6

What You Own vs What We Own

This is the most important difference between InfraReady and alternatives like Railway or Render. When you stop using InfraReady, your infrastructure keeps running — because it's yours.

The Customer Owns
All of this lives in your AWS account
  • VPC, subnets, NAT gateway — your private network
  • RDS PostgreSQL — your database, your data, your backups
  • ECS cluster, load balancer, ECR repository — your app hosting
  • S3 bucket, CloudFront CDN — your storage and static delivery
  • KMS encryption keys — you control your own encryption
  • CloudTrail audit logs — your compliance audit trail
  • GuardDuty, WAF, Macie, Inspector — your security tooling
  • All OpenTofu state files — you can manage infra directly anytime
  • AWS bills — infrastructure costs go directly to their AWS account
InfraReady Owns
The tooling — not the infrastructure
  • Lambda runner and SQS queue — our deployment engine
  • The web app (infraready.io) and its Vercel hosting
  • Supabase database — user accounts, project metadata, deployment records
  • The 11 OpenTofu modules (the blueprints — not their infrastructure)
  • GitHub App (InfraReady-app) — the integration tooling
  • ECR registry — stores our deployment Docker image only
  • Public S3 bucket — the CloudFormation template for customers to download
If a customer cancels: their VPC, RDS, ECS, and everything else keeps running in their AWS account. They just won't get future automated deployments. Zero lock-in.
